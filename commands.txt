Next question, how can I connect a custom app (I'm developing) to a local Psotgre SQL database in order to conduct testing?
To connect your Shopify custom app to a local PostgreSQL database for testing, you treat Postgres like any other backend dependency: your app’s backend server connects to it using a database library (Prisma, Knex, Sequelize, raw pg, etc.), and your App Home / Polaris UI just calls your backend’s endpoints.

This is fully compatible with the App Home setup described in App Home docs and the React Router / Remix templates.

Below is a step‑by‑step outline. I’ll assume:

You’re building a Node-based embedded app (React Router or Remix) like the Shopify React Router app template.
You have PostgreSQL running locally (either directly or via Docker).
1. Run a local PostgreSQL instance
You can either:

Option A: Native Postgres on your machine
Install and run Postgres:

# macOS (Homebrew example)
brew install postgresql
brew services start postgresql

# create a DB for your app
createdb my_shopify_app_dev
Option B: Docker
bash

docker run --name my-shopify-postgres \
  -e POSTGRES_USER=shopify \
  -e POSTGRES_PASSWORD=shopify \
  -e POSTGRES_DB=my_shopify_app_dev \
  -p 5432:5432 \
  -d postgres:16
Remember the connection details:

Host: localhost
Port: 5432
User: shopify
Password: shopify
Database: my_shopify_app_dev
2. Store DB credentials in environment variables
In your app folder, add to .env (or .env.local depending on your setup):

env
Copy
1
DATABASE_URL=postgres://shopify:shopify@localhost:5432/my_shopify_app_dev
Do not commit this file to Git (it should be in .gitignore).

Your Shopify app will already use .env for things like SHOPIFY_API_KEY, SHOPIFY_API_SECRET, etc., as described in App configuration & setup. You just add DATABASE_URL to that.

3. Pick a Node database library
Common choices:

Prisma (ORM with schema + migrations)
Knex.js (query builder)
Sequelize, TypeORM, or just the low-level pg driver
Below I’ll show Prisma (very common for Shopify apps), and then a simpler pg example.

3A. Example with Prisma
Install Prisma and the Postgres driver:
bash

In prisma/schema.prisma, set the datasource to Postgres:
prisma

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}
Define a test model (e.g. Campaign stored in your local DB):
prisma


model Campaign {
  id          Int      @id @default(autoincrement())
  shopDomain  String
  name        String
  status      String   @default("draft")
  createdAt   DateTime @default(now())
}
Run migrations & generate client:
bash
Copy
1
2
npx prisma migrate dev --name init
npx prisma generate
Create a Prisma client in your app (e.g. db.server.ts):

// app/db.server.ts (for Remix) or src/db.ts
import {PrismaClient} from '@prisma/client';

let prisma: PrismaClient;

declare global {
  // This is to reuse the client during dev with hot reload
  // eslint-disable-next-line no-var
  var __prisma: PrismaClient | undefined;
}

if (!global.__prisma) {
  global.__prisma = new PrismaClient();
}

prisma = global.__prisma;

export {prisma};

4. Use the DB from your Shopify app backend
How you expose this to your Polaris UI depends on your router (React Router vs Remix), but the pattern is the same:

Backend route hits Postgres.
Returns JSON.
App Home front-end calls that route and renders data with Polaris components.
I’ll show two examples:

Example: Remix route with Prisma (App Home)
If you’re using Remix as in the App Home docs, a route might be app/routes/app.campaigns.tsx:


// app/routes/app.campaigns.tsx
import type {LoaderFunctionArgs} from '@remix-run/node';
import {json} from '@remix-run/node';
import {useLoaderData} from '@remix-run/react';
import {authenticate} from '../shopify.server';
import {prisma} from '../db.server';

export async function loader({request}: LoaderFunctionArgs) {
  // Ensure this request comes from an authenticated Shopify admin session
  await authenticate.admin(request);

  const campaigns = await prisma.campaign.findMany({
    orderBy: {createdAt: 'desc'},
    take: 20,
  });

  return json({campaigns});
}

export default function CampaignsPage() {
  const {campaigns} = useLoaderData<typeof loader>();

  return (
    <s-page heading="Local campaigns">
      <s-section>
        <s-table variant="list">
          <thead>
            <tr>
              <th>Name</th>
              <th>Status</th>
              <th>Created</th>
            </tr>
          </thead>
          <tbody>
            {campaigns.map((c) => (
              <tr key={c.id}>
                <td>
                  <s-text type="strong">{c.name}</s-text>
                </td>
                <td>
                  <s-badge tone={c.status === 'active' ? 'success' : 'info'}>
                    {c.status}
                  </s-badge>
                </td>
                <td>{new Date(c.createdAt).toLocaleString()}</td>
              </tr>
            ))}
          </tbody>
        </s-table>
      </s-section>
    </s-page>
  );
}
This route:

Authenticates via authenticate.admin(request) (standard Shopify embedded setup)
Reads from your local Postgres DB via Prisma
Renders the data using Polaris web components like s-page, s-section, s-table, s-text, s-badge.
From App Home’s perspective, it doesn’t matter that the DB is local—it’s just your backend code.

Example: React Router + pg + JSON endpoint
If you’re using the React Router template, you might have an API route, e.g. server/routes/local-data.ts:


15

You do not connect Polaris directly to Postgres.

n that example, "records" is a file, the database or a folder?
In that example, records is just a JavaScript variable in memory – an array of objects in React state. It is not a file, not a folder, and not the database itself.

Here’s how all the pieces relate:

What records actually is
In the example:

jsx
Copy
1

records is a React state variable.
It holds an array of plain JS objects like:
js
Copy

    { id: 1, name: "Record A" },
    { id: 2, name: "Record B" },
    // ...
  ]
That data comes from your backend, via:
js
Copy
1
2
3
  const res = await fetch('/api/records');
  const data = await res.json();
  setRecords(data.records);
So the flow is:

Your browser (the App Home iframe) calls /api/records.
/api/records is an endpoint that you implement in your app server.
That endpoint queries your real database (e.g. Postgres) and returns JSON.
The React component receives data.records and stores it in the records state.
The JSX maps over records to render s-table rows.
At no point is records a file or folder. It’s just the client-side representation of what your backend returned.

Polaris components (like s-table, s-text, s-button) just render this records array.

Rephrasing in plain language
Think of records as a “list of items currently loaded in the UI”.

Those items originally come from your database, but by the time they are in records, they are just JS objects in React state.

When you create/update/delete:

The UI sends a request to your backend (POST /api/records, PUT /api/records/:id, DELETE /api/records/:id).
The backend changes the database.
Then the UI re-fetches data from the backend and updates records.
Quick mental model
Database: where data is stored (Postgres).
Backend API (/api/records): how the UI talks to the database.
records state: what the React component currently knows about the data, so it can draw the UI.
Polaris is only involved at the last step: rendering records in UI.
